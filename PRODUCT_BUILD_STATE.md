# Product Build State

Updated: 2026-07-23

## Objective

Build, verify, package, and integrate a production-quality native CodexFlow desktop GUI that runs on the user’s computer as the primary control surface, uses the existing local broker/runtime without Codex CLI execution, and reduces the legacy browser page to a deliberately minimal fallback.

## Product thesis

For developers working through a remote AI chat, CodexFlow is the most direct, spatial, and trustworthy way to operate multiple project-scoped coding sessions on their own computer.

## Current phase

Version 0.46.0 is released. It adds the equivalent Windows-native management and approval client: the full desktop information architecture, authenticated lifecycle and mutation APIs, Windows UI Automation Computer Use with app-only `PrintWindow` capture and identity-bound approvals, and an ephemeral WebView2 command runner with exact-origin navigation, screenshots, semantic DOM, sealed actions, exact-element comments, and bounded diagnostics. The one-command launcher retrieves a self-contained x64/arm64 release asset, verifies SHA-256, and installs it under `%LOCALAPPDATA%\CodexFlow`; no .NET setup is required. Main CI executes the WPF app and Computer helper on Windows x64 and builds verified release packages for x64 and arm64.

## Product truth

- CodexFlow is an independent local MCP broker that gives a user-owned ChatGPT conversation bounded access to approved repositories.
- One broker discovers local projects, serves one authenticated endpoint, and maintains independent project bindings for concurrent MCP conversations.
- ChatGPT supplies conversation and model reasoning. CodexFlow supplies projects, files, repository analysis, Git, guarded terminal execution, local environments, worktrees, instructions, skills, and handoff/context tools.
- Codex metadata is an optional, read-only project/history source. CodexFlow never starts, resumes, or performs work through the Codex CLI.
- Saved launch profiles persist under `~/.codexflow`; live transport sessions and activity are process-local and content-free.
- The native application controls and observes the local broker. It does not introduce a second chat client, model backend, or execution authority.
- SSH project routing requires a concrete locally approved alias and saved canonical folder. Every call revalidates destination and project fingerprints, root containment, symlinks, blocked paths, limits, secrets, and write/Bash policy; CodexFlow never installs or invokes Codex remotely.

## Interface residue

- The former `o` destination was a large authenticated browser dashboard that duplicated desktop responsibilities and felt like an admin document.
- Its obsolete renderer, browser profile editor, duplicate navigation, and terminal-first workflow were removed.
- Only authenticated API contracts and useful runtime truth were preserved; the replacement is a deliberately small recovery fallback.
- The public site remains an explanation surface and now describes the real native application.

## Architecture decisions

| Decision | Choice | Rationale | Consequence |
|---|---|---|---|
| Product relationship | Flow7 Endorsed | CodexFlow is an independent technical utility in the Flow7 family. | Flow7 supplies care, rhythm, typography, interaction, and provenance; CodexFlow owns its spatial expression. |
| Existing browser UI | Treat as interface residue | It is an admin-like duplicate of responsibilities that belong in the desktop product. | Rebuild it as a tiny authenticated fallback and deep-link surface. |
| Runtime boundary | Existing CodexFlow broker remains authoritative | The user explicitly rejected Codex CLI execution and does not need another backend. | The app is an authenticated observer/controller over existing local APIs and processes. |
| Native platform | SwiftUI on macOS; WPF on Windows | Each platform needs a real local approval surface and first-class OS browser/accessibility integration. | Deliver a universal macOS `.app` and self-contained Windows x64/arm64 assets with equivalent information architecture and broker contracts. |
| Installation | Package-shipped macOS app plus verified Windows release retrieval | Preserves the one-command experience without a setup wizard or separately installed desktop runtime. | macOS copies to `~/Applications`; Windows verifies an architecture-specific SHA-256 asset and installs to `%LOCALAPPDATA%\CodexFlow`. |
| Startup ownership | Native app may launch the existing Node broker using a non-secret desktop config | Enables useful offline launch and lifecycle controls while keeping one runtime implementation. | App starts `codexflow start --non-interactive --no-open-app`; logs remain local. |
| Runtime discovery | Read all private runtime records under `~/.codexflow/runtime` | Supports multiple projects/brokers and resumption without a registry service. | Tokens are held in memory and never rendered. |
| Native information architecture | Now / Projects / Environments / Worktrees / Changes / Tasks / Hosts / Computer / Browser / Connection / Policy | Maps directly to “what is live, where, what is progressing, how projects initialize, which checkouts exist, what changed, which hosts, apps, and origins are trusted, how to connect, what is allowed.” | No fake chat composer or redundant model experience. |
| Computer Use boundary | Platform-native helper plus broker-owned approvals | Native visual tasks need explicit local consent without allowing a model to bypass terminal, browser-host, or ChatGPT approval boundaries. | macOS uses Accessibility/Screen Recording and signing identity; Windows uses UI Automation, app-only `PrintWindow`, and publisher-or-binary identity. Snapshots expire and sensitive actions require exact local confirmation. |
| Session observability | Existing memory-only, content-free overview | Operators need route/health visibility without retaining prompts or repository content. | Show fingerprints, project, tool name/status/duration, and timestamps only. |
| Browser fallback | Authenticated status + native-app launch + emergency connection actions | Recovery paths still need a small local surface without duplicating the desktop app. | The `o` key opens the native app on macOS or Windows; the page remains manually reachable. |
| Browser feedback | Native exact-element selection with memory-only route comments | Human visual review belongs beside the rendered local page, while the model-visible payload must remain bounded and isolated. | Stable structural selectors omit page IDs/classes/data, password targets are ignored, comments are secret-checked and removed with the tab or revoked origin. |
| Browser diagnostics | Bounded route-private metadata rather than unrestricted CDP | Common page debugging needs console, resource timing, and source identity, but not cookies, storage, credentials, payload bodies, headers, or arbitrary browser-profile access. | The native and broker layers independently cap and sanitize diagnostics; URLs are query-free, secret-looking console messages are suppressed, and all state dies with the ephemeral tab. |
| Subagent ownership | ChatGPT Work owns models and child threads; CodexFlow owns local route coordination only | Preserves the user-required no-Codex/no-second-model-backend boundary while matching the Work execution model. | `agent_progress` can register only real parent-declared children, never launches a model, and grants each child a revocable project route. |
